# distill v0.1.1

> Any document → clean Markdown, in a single Go binary, built for LLM / RAG
> pipelines.

An incremental release that adds two capabilities on top of v0.1.0: fetching
input from a **URL / URI**, and an out-of-process **plugin system** for adding
formats without recompiling. No breaking changes — every v0.1.0 command,
flag, and surface behaves exactly as before.

---

## Upgrade — one command

```bash
curl -fsSL https://raw.githubusercontent.com/MitudruDutta/distill/main/install.sh | bash
```

To pin this version, set `DISTILL_VERSION=v0.1.1` before running the command.
Other paths (`go install`, container, source) are unchanged — see
[`docs/install.md`](https://github.com/MitudruDutta/distill/blob/main/docs/install.md).

## What's new

### URL / URI input

The one-shot form now accepts a URI as the first argument, so you can convert
remote documents without a separate download step:

```bash
distill https://example.com/whitepaper.pdf
distill http://example.com/data.csv -o data.md
distill 'data:text/csv;base64,YSxiCjEsMgo='
distill file:///etc/hosts
```

Supported schemes: `http`, `https`, `file`, `data`. Anything else is rejected.

HTTP(S) fetches are **SSRF-guarded by default**. The fetcher resolves DNS and
refuses to connect to loopback, RFC 1918 / RFC 4193 private ranges,
link-local — including the cloud-metadata IP **169.254.169.254** — and IPv6
loopback / link-local / unique-local addresses. The check runs on every
redirect hop, so DNS rebinding cannot bypass it. Other defaults: 30-second
total timeout, 32 MiB body cap, 5 redirect hops, and redirect targets must use
an allowed scheme.

Some sites reject the default `distill/0.1` agent; override it with
`--user-agent` or the `DISTILL_USER_AGENT` environment variable.

### Plugin system

distill can now hand formats it doesn't natively support to **out-of-process
plugins** — any executable that reads document bytes on stdin and writes
Markdown to stdout. Plugins are language-agnostic (a six-line shell script
works), with no cgo or Go-`plugin` constraints.

```bash
distill --list-plugins              # discover configured plugins
distill --use-plugins report.rtf    # enable for this conversion
DISTILL_USE_PLUGINS=1 distill x.rtf # enable via environment
```

Plugins are declared in `~/.config/distill/plugins.json` or a workspace
`./.distill/plugins.json`, discovered via a `--distill-capabilities`
handshake, and registered ahead of the built-ins (so a plugin can override a
format). They are **off by default**: running arbitrary executables is an
explicit opt-in. `batch`, `serve`, and `mcp` honor `DISTILL_USE_PLUGINS`.

Full protocol, manifest format, and a worked example:
[`docs/plugins.md`](https://github.com/MitudruDutta/distill/blob/main/docs/plugins.md).

## Security and robustness

- **SSRF protection** on URL fetches is on by default and re-checked on every
  redirect (see above).
- **Plugins are opt-in.** distill never executes a configured plugin unless
  you pass `--use-plugins` or set `DISTILL_USE_PLUGINS`.
- **Plugin execution is bounded.** Discovery (5 s) and conversion (120 s) are
  time-limited, and a plugin's stdout is capped (256 MiB) so a runaway or
  malicious plugin cannot exhaust memory; on overflow the child is killed
  rather than left to block. A forking plugin cannot wedge distill open past
  its timeout.

## Changelog since v0.1.0

**Features**

- URL / URI input for `http`, `https`, `file`, and `data` schemes, with
  SSRF protection.
- `--user-agent` flag and `DISTILL_USER_AGENT` for HTTP fetches.
- Out-of-process plugin protocol: `--use-plugins`, `--list-plugins`,
  `DISTILL_USE_PLUGINS`, manifest discovery, and registration ahead of
  built-in converters.

**Hardening**

- Cap plugin stdout and cancel a runaway plugin's context on overflow
  (memory- and time-bounded).

**Tests**

- URL fetching: SSRF (IPv4 / IPv6), error paths, malformed data URIs,
  timeouts, body-cap edge cases.
- Plugin runtime resilience: timeouts, forking children, the output-cap
  guard, stdin handling, binary round-trip, concurrency, and discovery
  isolation.

## Compatibility

No breaking changes from v0.1.0. New behavior is additive and off by default
(URL fetching activates only when the argument is a URI; plugins activate only
when explicitly enabled).
